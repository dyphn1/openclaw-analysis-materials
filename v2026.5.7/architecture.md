# OpenClaw v2026.5.7 架構分析

## 高層架構概覽

OpenClaw 的架構遵循模組化、插件化設計原則，核心概念包括：

1. **Gateway (控制平面)**：單一控制平面負責會話管理、工具執行、事件路由和插件協調。
2. **Agents (執行環境)**：隔離的執行環境，每個 agent 擁有自己的工作區、會話歷史和工具訪問權限。
3. **Channels (通訊介面)**: 透過插件系統連接到各種通訊平台（Discord、WhatsApp、Slack 等）。
4. **Plugins & Extensions**: 功能擴充機制，允許添加新的通訊頻道、工具或系統能力。
5. **Tools & Skills**: 代理可呼叫的能力，從基本的檔案操作到複雜的多步驟工作流程。

在 v2026.5.7 版本中，架構沒有發生根本性變更，但透過多個精準的修正增強了穩定性和正確性。特別是針對 Discord 外掛的訊息路由邏輯進行了修正，確保跨代理訊息發送時能正確區分頻道訊息和直接訊息。

## 核心架構元件

### Gateway 核心流程

Gateway 作為系統的中樞，其主要職責包括：

- **會話管理**: 建立、維護和終止 agent 會話
- **工具路由**: 將 agent 工具呼叫導向適當的實作（本地或沙箱）
- **事件分發**: 將進入的頻道事件分發到相關的 agent
- **插件協調**: 載入、配置和管理所有已安裝的插件
- **配置管理**: 讀取和應用全域和 per-agent 配置

在訊息處理方面，Gateway 遵循以下流程：

1. 進入訊息由特定頻道插件接收（例如 Discord 插件）
2. 插件將訊息轉換為標準化的 OpenClaw 訊息格式
3. Gateway 根據訊息的目標（session key）將其路由到適當的 agent 會話
4. Agent 處理訊息並可能透過工具發送回覆
5. 回覆訊息再次透過 Gateway 發送回原始頻道

### 插件架構

OpenClaw 的插件系統基於以下原則：

- **延遲載入**: 插件僅在需要時載入
- **介面標準化**: 所有插件實作標準的生命週期方法（setup、start、stop、teardown）
- **事件驅動**: 插件透過訂閱特定事件類型來響應系統事件
- **配置隔離**: 每個插件擁有獨立的配置命名空間

Discord 插件作為一個典型的頻道插件，實作了：

- **連接管理**: 建立和維持與 Discord API 的 WebSocket 連接
- **事件處理**: 將 Discord 事件轉換為 OpenClaw 內部事件格式
- **訊息發送**: 透過 Discord API 發送訊息到頻道或直接訊息
- **會話關聯**: 使用 session key 將 Discord 訊息與特定 agent 會話關聯

## 訊息路由深度分析

在 v2026.5.7 中，一個重要的修正是針對 Discord 訊息目標解析的修正。這個修正影響了訊息路由層，具體來說是：

### 問題描述

在 v2026.5.6 及更早版本中，當代理透過 `message(action="send")` 工具發送訊息時，如果目標是 `discord:channel:<id>`（頻道 ID），在特定上下文下（特別是當訊息來自直接訊息時），這個目標會被錯誤地解析為直接訊息目標，導致訊息發送失敗並回傳 "Unknown Channel" 錯誤。

### 根本原因

問題出現在 `session-key-normalization.ts` 模組中的 `normalizeExplicitDiscordSessionKey` 函數。當處理直接訊息上下文中的 `discord:channel:<id>` 目標時，函數會嘗試將其轉換為 `discord:direct:<id>`，假設發送者試圖回覆一個直接訊息。然而，在代理發送訊息的場景中，這個假設是錯誤的。

### 修正細節

修正後的邏輯現在會：

1. 只在會話鍵明確是 `discord:dm:` 格式時才進行直接訊息轉換
2. 保留 `discord:channel:<id>` 格式，除非能確認它確實對應於發送者的直接訊息 ID
3. 增加了發送者 ID 驗證：只有當提供的頻道 ID 實際匹配發送者的 Discord 用戶 ID 時，才會轉換為直接訊息

這個修正確保了當代理明確指定 `discord:channel:<id>` 作為目標時，訊息會正確路由到頻道，而不是被誤導為直接訊息。

## 配置與擴充點

### Discord 插件配置

Discord 插件透過以下配置鍵進行自訂：

- `channels.discord.token`: 機器人令牌（必要）
- `channels.discord.allowFrom`: 允許接收訊息的用戶/頻道允許列表
- `channels.discord.dmPolicy`: 私人訊息處理政策 (`pairing`、`open`、`block`)
- `channels.discord.enableVoice`: 是否啟用語音功能
- `channels.discord.audioQuality`: 語音音質設定

在 v2026.5.7 中，雖然沒有新增主要配置選項，但透過 `audit-core.ts` 的修正，增強了語音頻道權限的稽核，間接影響語音功能的行為。

### 擴充點

開發者可以透過以下方式擴充 OpenClaw 的功能：

1. **新增頻道插件**: 實作 `src/plugins` 中的介面
2. **添加自定義工具**: 在 `src/agents/tools/` 中創建新的工具實作
3. **擴充技能系統**: 透過 ClawHub 分享或建立新的技能
4. **自定義中介軟體**: 在 Gateway 配置中添加請求/響應攔截器

## 版本演變與向後相容性

v2026.5.7 保持完全的向後相容性：

- 沒有移除或廢棄任何公開介面
- 所有修正都是錯誤修復，不改變既有功能的預期行為
- 配置schema保持不變
- 插件介面保持穩定

這個版本的重點是透過精準的錯誤修復提升系統穩定性，特別是在跨代理通訊和頻道特定功能方面。

## 總結

OpenClaw v2026.5.7 版本代表了對現有架構的穩定性和正確性改進。透過針對 Discord 訊息路由的精準修正，以及其他多個子系統的錯誤修復，該版本提升了系統在真實世界使用中的可靠性。架構保持模組化和可擴充，為未來的功能增強奠定了堅實的基礎。
